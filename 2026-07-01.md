## 📝 技术笔记：LLM Agent 架构设计：ReAct、Plan-and-Execute、Multi-Agent

好的，这是关于LLM Agent架构设计的技术笔记。

在构建复杂的LLM应用时，单一的提示词调用往往力不从心。我们需要更系统化的“智能体”架构，赋予大模型规划、使用工具和多轮推理的能力。本文将探讨三种主流范式：ReAct、Plan-and-Execute以及Multi-Agent。

### 1. ReAct: 思考与行动交织

**核心概念**：ReAct (Reasoning + Acting) 让大模型在解决问题时，交替进行“推理”（Thought）和“行动”（Action）。模型首先生成一段关于当前情况和下一步计划的推理文本，然后基于这个推理决定调用哪个工具（如搜索引擎、计算器），获取观察（Observation）结果后，再继续下一轮推理。这种“思考-行动-观察”的循环是其核心。

**技术原理**：ReAct 将推理步骤显式化，使模型的行为轨迹更透明、可控。它通过一个循环来驱动：
```python
# 伪代码：ReAct 循环核心
prompt = f"问题: {question}\n历史:"
while not is_finished:
    # 1. 生成思考和行动
    thought_action = llm.generate(prompt + "\nThought:")
    thought, action = parse_thought_action(thought_action)
    
    # 2. 执行行动，获取观察
    if action.type == "ToolCall":
        observation = execute_tool(action.tool_name, action.input)
        prompt += f"\nThought: {thought}\nAction: {action}\nObservation: {observation}"
    elif action.type == "Finish":
        answer = action.input
        is_finished = True
```

**应用场景**：适合需要动态决策和信息检索的复杂问答任务。例如，一个研究助手Agent，它需要根据用户问题动态决定是搜索最新资料，还是查询数据库，或是进行数学计算，并将中间结果用于后续推理。

**常见坑点**：
*   **循环与发散**：模型可能在不相关的工具调用间循环，或思考方向错误却无法自我纠正，需要设置最大循环次数和清晰的终止条件。
*   **提示词长度膨胀**：每一步的思考、行动和观察都会累积进上下文，可能导致最终超出模型的上下文窗口限制，需设计摘要或压缩机制。

### 2. Plan-and-Execute: 先规划，后执行

**核心概念**：该模式将任务解耦为两个阶段。首先由一个“规划器”制定一个高层级的、可执行的步骤列表；然后由一个“执行器”逐步执行这些计划中的每一步，执行器可以是一个专门的Agent或一系列工具链。

**技术原理**：它模仿人类处理复杂项目的方式——先有蓝图，再分步施工。
```python
# 伪代码：Plan-and-Execute 流程
plan = planner_llm.generate(f"请为完成任务制定一个步骤计划: {task}")
steps = parse_plan(plan) # 解析为步骤列表

results = []
for i, step in enumerate(steps):
    # 执行器可以是另一个LLM调用，也可以是工具链
    result = executor.run(f"根据之前的步骤结果 {results} 和当前任务 {step}，请执行。")
    results.append(result)
    # （可选）根据执行结果，计划可能被动态调整
    if needs_replanning(results):
        plan = planner_llm.replan(plan, results)
        steps = parse_plan(plan)
```

**应用场景**：非常适合目标明确、步骤可预先分解的任务。例如，“写一篇关于AI Agent的技术博客并发布”，规划器可能生成大纲、撰写各部分、添加代码、格式化、发布的步骤；执行器依次完成。

**常见坑点**：
*   **计划僵化**：初始计划可能基于错误假设。如果执行过程中发现新信息，计划调整（Re-planning）的机制设计至关重要。
*   **规划器与执行器的能力错配**：规划器需要强大的逻辑推理和任务分解能力，执行器则需要精准执行特定子任务。两者需要匹配。
*   **错误累积**：前序步骤的执行错误会被后续步骤依赖，导致雪崩式失败，需要设计检查点或回滚机制。

### 3. Multi-Agent: 群体协作

**核心概念**：引入多个具备不同角色、能力或视角的Agent，它们通过通信（如聊天、共享黑板、消息队列）进行协作，以完成单一Agent难以实现的复杂目标。

**技术原理**：常见架构包括中心化的“指挥官-工作者”模式，或去中心化的“平等协商”模式。核心是角色定义与通信协议。
```python
# 伪代码：一个简化的“研究员-作家”多智能体协作
researcher_agent = create_agent(role="研究员", tools=[search, arxiv])
writer_agent = create_agent(role="技术作家", tools=[code_interpreter, writer])

# 协作过程（简化为对话）
task = "撰写一篇关于最新注意力机制的技术综述"
discussion_history = []

# 研究员先工作
research_output = researcher_agent.execute(f"请针对任务 {task} 进行深度调研并总结关键点。")
discussion_history.append(f"研究员: {research_output}")

# 作家基于调研结果进行创作
article_draft = writer_agent.execute(
    f"基于以下调研结果，请撰写技术综述文章。\n调研结果: {research_output}\n当前讨论记录: {discussion_history}"
)
discussion_history.append(f"作家: {article_draft}")

# 可能还需要审阅、反馈、修改的多轮循环
```

**应用场景**：适用于模拟真实团队工作流程的复杂项目。例如，软件开发（产品经理、架构师、程序员、测试Agent）、辩论与决策系统（多个具有不同立场的Agent探讨问题）、角色扮演游戏（NPC自主交互）。

**常见坑点**：
*   **通信开销与死锁**：Agent间通信可能产生大量冗余信息，消耗计算资源和上下文空间。设计不良的协议可能导致任务停滞。
*   **角色混乱与越界**：如果没有严格的职责边界，Agent可能做出不符合其角色设定的行为，导致混乱。
*   **成本与延迟倍增**：多轮多Agent交互意味着更多的LLM API调用，导致成本和延迟显著增加，需权衡收益。
*   **评估困难**：对最终输出的归因和调试变得复杂，难以判断是哪个环节或哪个Agent的决策出了问题。

**总结**：ReAct是动态反应式的“单人探险家”；Plan-and-Execute是目标驱动的“项目经理”；Multi-Agent是角色分工的“项目团队”。选择哪种架构，取决于任务复杂度、对动态性的要求、成本容忍度以及对过程透明度的需求。实践中，这些架构常被混合使用，例如在Multi-Agent系统中，每个“工作者”Agent内部可能就采用ReAct模式。

# 使用Python实现简单的语义搜索

基于`sentence-transformers`库，我们可以快速构建一个轻量级的语义搜索系统。这个示例将文本转换为向量，通过计算余弦相似度来查找最相关的文档，适用于问答系统、知识库检索等场景。

```python
from sentence_transformers import SentenceTransformer
import numpy as np
from typing import List, Tuple

class SemanticSearcher:
    """基于sentence-transformers的语义搜索器"""
    
    def __init__(self, model_name: str = 'all-MiniLM-L6-v2'):
        """
        初始化语义搜索器
        
        Args:
            model_name: 预训练模型名称，默认使用all-MiniLM-L6-v2
        """
        self.model = SentenceTransformer(model_name)
        self.documents: List[str] = []
        self.document_embeddings: np.ndarray = None
    
    def add_documents(self, documents: List[str]) -> None:
        """
        添加文档并计算其向量表示
        
        Args:
            documents: 待搜索的文档列表
        """
        self.documents.extend(documents)
        new_embeddings = self.model.encode(documents)
        
        if self.document_embeddings is not None:
            self.document_embeddings = np.vstack([self.document_embeddings, new_embeddings])
        else:
            self.document_embeddings = new_embeddings
    
    def search(self, query: str, top_k: int = 3) -> List[Tuple[int, str, float]]:
        """
        执行语义搜索
        
        Args:
            query: 查询文本
            top_k: 返回结果的数量
            
        Returns:
            包含(索引, 文档, 相似度分数)的列表
        """
        if self.document_embeddings is None or len(self.documents) == 0:
            raise ValueError("请先添加文档")
        
        query_embedding = self.model.encode([query])[0]
        similarities = np.dot(self.document_embeddings, query_embedding) / (
            np.linalg.norm(self.document_embeddings, axis=1) * np.linalg.norm(query_embedding)
        )
        
        # 获取最相似文档的索引
        top_indices = np.argsort(similarities)[-top_k:][::-1]
        
        results = []
        for idx in top_indices:
            results.append((idx, self.documents[idx], float(similarities[idx])))
        
        return results

# 使用示例
if __name__ == "__main__":
    # 创建搜索器实例
    searcher = SemanticSearcher()
    
    # 添加示例文档
    documents = [
        "Python是一种广泛使用的高级编程语言",
        "机器学习是人工智能的一个子领域",
        "深度学习使用神经网络处理复杂数据",
        "自然语言处理使计算机能够理解人类语言",
        "计算机视觉专注于图像和视频分析"
    ]
    searcher.add_documents(documents)
    
    # 执行搜索
    query = "什么是人工智能技术？"
    results = searcher.search(query, top_k=3)
    
    print(f"查询: {query}")
    print("搜索结果:")
    for idx, doc, score in results:
        print(f"{idx+1}. {doc} (相似度: {score:.3f})")
```

## 说明
1. **SentenceTransformer**：使用预训练模型将文本转换为向量表示，语义相似的文本在向量空间中更接近。
2. **余弦相似度**：通过计算向量夹角的余弦值来衡量文本相似度，值域为[-1,1]，越接近1表示越相似。
3. **扩展建议**：对于大规模文档，可以考虑使用FAISS等向量数据库优化检索性能。
