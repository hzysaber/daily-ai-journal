## 📝 技术笔记：大模型微调实战：LoRA、QLoRA、全量微调的适用场景

## 核心概念

大模型微调（Fine-tuning）是指在预训练模型的基础上，使用特定领域或任务的数据继续训练，使模型获得针对性能力。根据参数更新策略和资源消耗，主流方案可分为三类：

| 方案 | 可训练参数占比 | 显存需求（7B模型） | 训练速度 | 适用场景 |
|------|---------------|-------------------|----------|----------|
| **全量微调** | 100% | ~60GB（FP16） | 慢 | 数据充足、追求极致效果 |
| **LoRA** | 0.1%-1% | ~16GB | 快 | 通用场景、资源有限 |
| **QLoRA** | 0.1%-1% | ~6GB | 中 | 消费级GPU、实验探索 |

## 技术原理

### 全量微调（Full Fine-tuning）

直接更新模型所有参数。本质是标准的反向传播，梯度贯穿整个模型。优势在于表达能力不受限，但计算和存储开销巨大，且存在灾难性遗忘风险。

### LoRA（Low-Rank Adaptation）

核心思想：冻结原始权重 $W_0$，用两个低秩矩阵 $A \in \mathbb{R}^{d \times r}$、$B \in \mathbb{R}^{r \times k}$ 近似增量更新：

$$W = W_0 + \Delta W = W_0 + BA$$

其中 $r \ll \min(d,k)$，通常取 8-64。参数量从 $d \times k$ 降至 $(d+k) \times r$，可训练参数减少 100-1000 倍。

### QLoRA

在 LoRA 基础上引入量化技术：将基础模型量化为 4-bit（NF4格式），仅对 LoRA 适配器保持高精度训练。配合**双重量化**和**分页优化器**，7B 模型可在单张 24GB 显卡上训练。

## 实际应用场景

**选 LoRA/QLoRA 的场景：**
- 垂直领域问答（医疗、法律、金融）
- 特定风格对话生成
- 多语言能力增强（如中文补强）
- 快速原型验证、AB 测试不同数据集效果
- 需要维护多个细分模型（LoRA 适配器仅几十 MB，可热插拔）

**选全量微调的场景：**
- 基座模型→指令模型的对齐训练（SFT阶段）
- 模型需要根本性改变（如大幅调整词表、切换领域）
- 训练数据量级达到百万级以上
- 硬件资源充足（多卡/A100集群）

## 常见坑点

1. **LoRA rank 选择**：r 不是越大越好。任务简单时 r=8 足够，过大的 r 反而过拟合。建议从 r=16 开始，根据验证集 loss 调整。

2. **target_modules 遗漏**：只对 attention 的 Q/V 投 LoRA 效果有限。推荐同时覆盖 QKV + MLP 层，效果显著提升。

3. **学习率陷阱**：LoRA 学习率通常比全量微调高一个数量级（1e-4 ~ 3e-4），全量微调建议 1e-5 ~ 5e-5。用错会导致不收敛或震荡。

4. **QLoRA 的 bfloat16 问题**：部分老卡（如 V100）不支持 bf16，需改为 fp16，否则报错。

5. **梯度检查点**：显存紧张时务必开启 `gradient_checkpointing=True`，可节省 30%-40% 显存，代价是训练速度下降约 20%。

6. **合并权重后精度下降**：LoRA 合并（merge）时如果 base model 和 adapter 精度不一致（如 fp16 base + fp32 adapter），可能出现数值抖动。建议统一为 fp16 后再合并。

7. **数据质量 > 微调技巧**：1000 条高质量数据的微调效果，往往优于 10 万条噪声数据。数据清洗、去重、格式统一是前置必做项。

## 💻 代码片段：用Python实现一个简单的重试装饰器（支持指数退避）

在Python中，处理网络请求或数据库操作时经常需要重试机制。下面是一个简洁实用的重试装饰器实现，支持指数退避策略，能有效避免频繁请求导致的限流问题。

```python
import time
from functools import wraps
from typing import Callable, Type, Tuple, Optional, Any

def retry_with_backoff(
    max_retries: int = 3,
    base_delay: float = 1.0,
    exponential_base: float = 2.0,
    exceptions: Tuple[Type[Exception], ...] = (Exception,),
    on_retry: Optional[Callable[[int, Exception], None]] = None
) -> Callable:
    """
    重试装饰器，支持指数退避策略
    
    参数:
        max_retries: 最大重试次数（默认3次）
        base_delay: 初始延迟时间（秒，默认1.0）
        exponential_base: 指数退避基数（默认2.0）
        exceptions: 需要重试的异常类型元组（默认捕获所有异常）
        on_retry: 重试时的回调函数，参数为(retry_count, exception)
    
    用法:
        @retry_with_backoff(max_retries=5)
        def unstable_function():
            # 可能失败的代码
            pass
    """
    def decorator(func: Callable) -> Callable:
        @wraps(func)
        def wrapper(*args: Any, **kwargs: Any) -> Any:
            last_exception = None
            
            for attempt in range(max_retries + 1):
                try:
                    return func(*args, **kwargs)
                except exceptions as e:
                    last_exception = e
                    
                    if attempt == max_retries:
                        raise e
                    
                    # 计算延迟时间（指数退避 + 随机抖动）
                    delay = base_delay * (exponential_base ** attempt) * (0.5 + 0.5 * hash(str(e)) % 100 / 100)
                    
                    # 调用重试回调
                    if on_retry:
                        on_retry(attempt + 1, e)
                    
                    time.sleep(delay)
            
            raise last_exception  # 不会执行到这里，但为了类型检查
        
        return wrapper
    return decorator


# 使用示例
@retry_with_backoff(
    max_retries=4,
    base_delay=0.5,
    exceptions=(ConnectionError, TimeoutError),
    on_retry=lambda attempt, e: print(f"重试第{attempt}次，错误: {e}")
)
def fetch_data(url: str) -> str:
    """模拟可能失败的网络请求"""
    import random
    if random.random() < 0.7:  # 70%概率失败
        raise ConnectionError(f"无法连接到 {url}")
    return f"成功获取 {url} 的数据"


if __name__ == "__main__":
    try:
        result = fetch_data("https://example.com")
        print(f"结果: {result}")
    except Exception as e:
        print(f"最终失败: {e}")
```

这个装饰器通过指数退避策略智能地增加重试间隔，能有效应对临时性故障。它支持指定异常类型、自定义重试次数，并提供回调函数监控重试过程。通过随机抖动还可以避免多个客户端同时重试导致的"惊群效应"。
